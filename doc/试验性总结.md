##### 1、假设某块磁盘的参数如下：

容量为 1TB，磁盘转速为 7200RPM，最大寻道时间为20ms，平均寻道时间为 5ms，最小寻道时间为 0.5ms（指磁头寻道到相邻磁道的时间），一个磁道大小为 64MB。如果磁盘块大小为 16KB，请回答下面问题（所有结果均四舍五入保留小数点后 1 位）：

###### （1）

> 磁盘块读取时间=寻道时间+旋转延迟+传输时间+其他延迟，其中，传输时间可根据磁盘转速和磁道与块的比例大小换算，而其他延迟典型情况下可约为0。

传输时间：（1/7200PPM）*（16KB/64MB）=0.002ms

则随机读取一个磁盘块的平均延迟是 5.00+4.17+0.002$\approx$9.2ms

##### 2、目前不同的 DBMS 所采用的磁盘块大小往往不同，

例如 MySQL 是 16KB，而 MS SQL Server 是 8KB。其它的一些系统如 HDFS 则采用了默认 64MB 的大块。此外，像 DB2、 Oracle 等还支持可变的磁盘块大小（比如从 8KB 到 64KB），但有的 DBMS（如 MS SQL Server）却采用固定的磁盘块大小（8KB）。针对这些现象，请回答下面的问题：

（1） DBMS 采用固定磁盘块大小和可变磁盘块大小这两种策略各有什么优缺点？

（2） 目前流行的关系型 DBMS 为什么不采用 64MB 以上的大块设计？

（3） 什么样的数据库应用适合在关系型 DBMS 中采用 64MB 这样的大块设计？试结合一两个例子给出你的分析。

###### （1）

- 固定磁盘块大小
  - 优点
    - 简单性：管理和实现较为简单，因为所有快的大小相同，便于内存和磁盘之间的数据传输
    - 一致性：固定大小的块使得磁盘空间的分配和管理更加一致，减少了碎片化的可能性
    - 性能优化：可以针对特定的块大小进行性能优化，便于缓存和预取策略
  - 缺点
    - 空间浪费：如果记录的大小不一致，可能导致空间浪费。例如，小记录可能占用整个块，而大记录可能需要多个块
    - 灵活性不足：无法根据不同类型的数据调整块大小，可能影响性能和存储效率
- 可变磁盘块大小
  - 优点
    - 灵活性：可以根据不同类型的数据和访问模式调整块大小，提高存储和访问效率
    - 空间利用率高：可以更好地适应不同大小的记录，减少空间浪费
  - 缺点
    - 复杂性：管理和实现较为复杂，需要处理不同大小的块，系统较为复杂
    - 碎片化：可能导致磁盘碎片化，影响性能和存储效率
    - 性能优化难度大：由于块大小不固定，针对特定块大小的性能优化策略可能无法有效应用

###### （2）

1. 内存和缓存管理：大块设计会占用大量内存，尤其在高并发环境下，可能导致内存资源快速耗尽；较小的块大小可以高效地利用系统缓存，因为缓存中存储更多的块，从而提高缓存命中率
2. I/O性能：随机I/O时，较大的块大小会导致每次I/O操作需要读取或写入更多的数据，增加了I/O延迟；同时，较小的块才能更好地支持并行I/O操作
3. 数据局部性：较小的块更好地利用了局部性原理，提高查询效率；较大的块会导致每次读写传输大量不必要的数据，增大开销
4. 碎片化和空间利用：较大块可能导致更多内部碎片化，浪费存储空间；较小的块可以灵活地分配和管理存储空间，提高空间利用率
5. 事务和并发控制：较小的块提供更细粒度的锁定机制，提高并发控制的效率；较小的块有助于更精细地控制事务隔离级别，减少事务冲突
6. 恢复和备份：较小块可以更快速地进行数据恢复，因为每次恢复操作只需要处理较小的数据块；较小的块大小可以更高效地进行增量备份和恢复。

###### （3）

1. 数据仓库和大数据分析：通常需要处理大量的批量数据加载和复杂的查询操作。在这些场景中，数据访问模式通常是顺序读取和批量处理，而不是随机访问。大块设计可以显著提高顺序读取的效率，因为每次读取操作可以获取更多的数据，减少了磁盘寻道时间。：大块设计适合批量数据处理，可以一次性加载大量数据，提高数据加载和处理的效率。
2. 多媒体存储和流媒体服务：需要存储和传输大量的音频、视频和图像数据。这些数据通常是大文件，并且访问模式主要是顺序读取。多媒体文件通常是大文件，采用大块设计可以减少文件存储和读取的碎片化，提高存储和读取的效率。多媒体文件的访问模式主要是顺序读取，大块设计可以减少磁盘寻道时间，提高顺序读取的效率。

##### 3、假设磁盘块大小为 8 KB，

块中存储 200 字节的定长记录，块首部只包括一个 8 字节的模式指针和一个偏移量表。对于插入块内的每条记录，在偏移量表中都增加一个 2 字节的指针指向该记录。假设每天向块内插入 4 条记录（空间不足时允许插入部分记录后结束全部操作），删除 2 条记录。假设每天的删除记录操作总是发生在插入记录之前，删除记录使用一个“删除标记”代替记录在偏移量表中的指针。给定一个磁盘块，如果刚开始块是空的，则几天后不能再向该块内插入记录？此时，该块内一共有多少条记录？

> 第一天，磁盘块无记录，删除操作失败，直接插入4条记录，占用空间大小（200+2）* 4=808B
>
> 此后（删除标记对应记录空间可覆写），每天删除2条记录，释放空间200*2=400B，插入4条记录，占用空间808B，即808-400=408B

所以，808+408 * x <=8 * 1024-8，即 x<=18；

因此，第x+1=19天插入记录后，占用空间808+408 * 18=8152，共4+2*18=40条记录，剩余8 * 1024-8-8152=32B空间；

最后，第20天，删除两条记录，剩余32+400=432B可用空间，再插入两条记录，剩余432-404=28B空间，此时可再插入一条部分记录，偏移量表占用2B，部分记录占用26B，但这里不认为是有效记录，即共40-2+2=40条记录；

综上，20天后不能再向该块内插入记录；此时，该块内一共有40条记录。

##### 5、我们在讲义上介绍了 SSD 感知的 CF-LRU 算法，

即 Clean-First LRU 算法。该算法虽然看起来可以减少对 SSD 的写操作，但依然存在一些问题。请分析一下该算法的主要缺点有哪些？给出两点，并简要解释你的理由。

1. 可能导致脏页积累：由于优先替换干净页，脏页可能会在缓冲区中积累。如果缓冲区中脏页过多，最终还是需要将这些脏页写回到 SSD，这可能会导致突发的大量写操作，影响系统的性能和稳定性。此外，脏页的积累也可能导致缓冲区空间不足，影响缓存命中率。
2. 可选干净页的Windows的大小不好控制，需要根据整个系统的数据置换大小而定
3. 增加了读操作的复杂性：选择要替换的页时优先选择干净页，这意味着在缓冲区中需要维护额外的信息来区分干净页和脏页。这种额外的信息维护和判断会增加读操作的复杂性和开销，可能导致整体性能下降。

##### 8、对于 B+树，假设有以下的参数：

| **参数** | **含义**                     | **参数** | **含义**                                             |
| -------- | ---------------------------- | -------- | ---------------------------------------------------- |
| N        | 记录数                       | S        | 读取一个磁盘块时的寻道时间                           |
| n        | B+树的阶，即节点能容纳的键数 | T        | 读取一个磁盘块时的传输时间                           |
| R        | 读取一个磁盘块时的旋转延迟   | m        | 在内存的 m 条记录中查找 1 条记录的时间  （线性查找） |

假设所有磁盘块都不在内存中。现在我们考虑一种压缩 B+树，即对 B+树的节点键值进行压缩存储。假设每个节点中的键值压缩 1 倍，即每个节点在满的情况下可压缩存储 2n 个压缩前的键值和 2n+1 个指针。额外代价是记录读入内存后必须解压，设每个压缩键值的内存解压时间为 c。给定 N 条记录，现使用压缩 B+树进行索引，请问在一棵满的 n 阶压缩 B+树中查找给定记录地址的时间是多少？（使用表格中的参数表示，n＋1 或 n-1 可近似表示为 n）？

> 根据B+树结构，逐层读入节点块到内存，在内存中索引下一次层节点，因此需要B+树深度次磁盘读取和解压缩，以及线性查找

假设B+树深度为d，则非叶子节点对记录进行了d-1次扩展，每次扩展2n+1次，最后每个叶子节点存储2n条记录，即：
$$
(2n+1)^d-1 \times 2n=N
$$
则B+树深度为：
$$
d=log_{2n}N
$$
因此，需要进行d次磁盘块读取、块解压缩、块线性查找，即在一棵满的 n 阶压缩 B+树中查找给定记录地址的时间是：
$$

(R+S+T+2nc+2n)\times log_{2n}N
$$
